### Isolated Infrastructure & Network Routing

Security and environment predictability are maintained through isolated Docker bridge networks, preventing unauthorized cross-network access and isolating running agent processes from direct host system access.

---

## Network Architecture & Topologies

Downstream tools and agent containers are split into distinct, isolated Docker bridge networks:

- **`tupynambalucas-agents-net`**: Dedicated sandbox network where AI agent containers (e.g., Google Antigravity or GitHub Copilot CLI terminals) execute commands. This prevents tasks from accessing other internal tools directly.
- **`tupynambalucas-mcp-net`**: Internal network housing downstream Model Context Protocol (MCP) server containers (e.g., GitHub, Firecrawl, or Context7 APIs).
- **`tupynambalucas-platform-net`**: Platform-wide network hosting the central Go-based ingress gateway.

---

## AgentGateway Routing Configuration

The **AgentGateway** (located in the `/platform` workspace) acts as the single entrypoint for all Model Context Protocol (MCP) clients and federates requests across the subnetworks:

- **Go-Based Ingress Proxy**: A high-performance Go-based federation proxy serves as the entrypoint on the host machine. It routes and aggregates commands sent to downstream MCP containers.
- **Network-Level CORS & SSE Handling**: The gateway intercepts preflight requests and injects appropriate CORS headers globally. It disables default HTTP timeouts on proxies to prevent active Server-Sent Events (SSE) connections from dropping during long-running tasks.
- **Internal Domain Routing**: Within the MCP stack, the gateway is aliased internally. Agent containers running in separate network namespaces route requests directly through the gateway via the platform network bridge.
- **Host-to-Container Bridge**: Local development clients communicate with the gateway from the host machine loopback interface using clean API endpoints (e.g., `/github/sse`, `/firecrawl/sse`).
- **Buffered stdio Stream Mapping**: The adapter wrapper executes child processes locally, preventing stdout fragmentation. By checking JSON-RPC transaction `id`s line-by-line, the adapter resolves HTTP requests with complete, un-truncated payloads.
