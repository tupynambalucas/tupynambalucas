# Infrastructure Standards

To keep the **Elo Orgânico** infrastructure, DevOps tooling, and background runtime environments robust, maintainable, and highly modular, this document unifies our self-hosted infrastructure standards, directory conventions, and multi-service Docker topologies.

---

## 1. Docker Compose Strategy (Dev vs. Production)

Designed for **Self-Hosted Excellence** on Hetzner Cloud.

Each bounded context (e.g., `instance/`, `portal/`) maintains a modular Compose configuration using environment overrides/inheritance:

| File                    | Environment          | Services                                    | Database               |
| :---------------------- | :------------------- | :------------------------------------------ | :--------------------- |
| `compose.yaml`          | Base Configuration   | API + Web + Redis + MongoDB + Init          | -                      |
| `compose.override.yaml` | Development (Local)  | MongoDB Replica Set + Redis (Build & Ports) | Local Docker container |
| `compose.prod.yaml`     | Production & Staging | API + Web + Redis (Build/Limits/Replicas)   | MongoDB Atlas (cloud)  |

For instructions on starting these environments, read the [Command Reference](../commands-reference.mdx).

The `compose.prod.yaml` has no MongoDB service by design. In production and staging, the API connects to a cloud-hosted MongoDB Atlas cluster via URI injected from the environment-specific `.env.prod` or `.env.staging` file.

### 1.1. Project and Container Naming Conventions

To prevent naming collisions on hosts that run multiple stacks or environments concurrently (such as staging and production on the same server), the following conventions are enforced:

- **Project Names**: Dev compose files define `name: elo-[context]-dev` (e.g., `elo-instance-dev`), while production/staging compose files define `name: elo-[context]-prod` (e.g., `elo-instance-prod`).
- **Dev Environments**: Container names carry an explicit `-dev` suffix (e.g., `elo-instance-redis-dev`, `elo-instance-db-dev`).
- **Staging vs. Production**: Production/staging containers dynamically append the environment name suffix using the `APP_ENV` variable (e.g., `container_name: elo-instance-redis-${APP_ENV:-prod}`). This isolates the containers at the Docker host daemon level, allowing staging and production containers to coexist on the same virtual machine without collisions.

### 1.2. Why Production and Staging Share a Single Compose File

The decision to use one `compose.prod.yaml` for both production and staging is a deliberate architectural choice, not a simplification shortcut. The rationale is grounded in the **Single Responsibility Principle (SRP)**:

A Compose file should have exactly one reason to change: a **change in stack topology** (new services, different network configuration, altered healthcheck intervals, additional containers). Neither production nor staging requires a different service topology — both run the identical set of services (API, Web/Nginx, Redis) connected to the identical network and with the identical healthcheck strategy.

The only differences between the two environments are **configuration values** (MongoDB Atlas cluster URI, Cloudflare Turnstile keys, API secrets, image tags). This is a configuration concern, not a topology concern, and is the sole responsibility of the environment-specific `.env` files (`.env.prod`, `.env.staging`).

Creating a separate `compose.staging.yaml` as a structural copy of `compose.prod.yaml` would violate SRP by introducing two files with the same reason to change. Any future topology change (adding a monitoring sidecar, updating healthcheck intervals) would require identical edits in both files, creating a drift risk. The correct segregation level is the `--env-file` flag, which is the Docker-sanctioned mechanism for environment-specific configuration injection.

## 2. Dockerfile Architecture

Each application owns its `Dockerfile` co-located within its directory, following the **Single Responsibility** and **Bounded Context** principles:

```plaintext
instance/services/api/Dockerfile    # Multi-stage: turbo prune → build → pnpm deploy → runner
instance/services/web/Dockerfile    # Multi-stage: turbo prune → vite build → nginx:alpine
portal/apps/api/Dockerfile      # Same pattern as instance/api
portal/apps/web/Dockerfile      # Same pattern as instance/web
```

Shared database and cache services (MongoDB, Redis) have their Dockerfiles and local environment files encapsulated inside `instance/services/mongodb/` and `instance/services/redis/` respectively, keeping them completely self-contained.

The API Dockerfiles use **Turborepo pruning** (`turbo prune`) and **`pnpm deploy`** to produce minimal images — no source code, no devDependencies, no monorepo symlinks in production.

The Web Dockerfiles use a two-stage build: a Node.js builder compiles the React SPA via Vite (with `VITE_*` variables injected as Docker `ARG`), and the final stage serves the static output from `nginx:alpine` (~25MB image).

## 3. Environment Variable Strategy

Environment variables follow the **Bounded Context** and **Single Responsibility** principles. To eliminate redundancy and prevent sync drift, all environment configuration files are centralized under the orchestrator scope (`[context]/infrastructure/docker/`). Services are kept completely stateless and load their processes via process-level injection:

| File Pattern                                     | Owner / Target   | Purpose / Scope                                                                        | Git         |
| :----------------------------------------------- | :--------------- | :------------------------------------------------------------------------------------- | :---------- |
| `[context]/infrastructure/docker/.env.dev`       | Local Dev Stack  | Centralized variables for development (API host config, local DB/Redis credentials)    | ignored     |
| `[context]/infrastructure/docker/.env.prod`      | Production Stack | Centralized production secrets, MongoDB Atlas URIs, Turnstile production credentials   | ignored     |
| `[context]/infrastructure/docker/.env.staging`   | Staging Stack    | Centralized staging configs, MongoDB Atlas staging URIs, Turnstile staging credentials | ignored     |
| `[context]/infrastructure/docker/.env.*.example` | Repository       | Onboarding configuration templates for dev, prod, and staging deployment scopes        | **tracked** |

_Note: In the table above, `[context]` represents either the `instance` or `portal` directory._

`VITE_*` variables (e.g., `VITE_TURNSTILE_SITE_KEY`) are **build-time** only — Vite bakes them statically into the JavaScript bundle during `vite build`. In Docker builds, they are passed via `ARG` → `ENV` before the build step runs. In local development, Vite loads variables directly from the centralized `[context]/infrastructure/docker/.env.dev` directory via its `envDir` configuration.

## 4. Network & Latency

All components in a deployment stack reside in a private named Docker bridge network (e.g., `elo_instance_network`) for sub-millisecond latency between the API and Redis. The Nginx container is the only service with an external port binding; the API is only reachable via the Nginx reverse proxy on the internal network.

---

## 5. Nginx Configuration

The Web containers (Nginx) handle two concerns:

1. **SPA Routing**: All unknown paths fall back to `index.html` via `try_files $uri $uri/ /index.html`, allowing React Router to handle client-side navigation without 404s on page refresh.
2. **API Reverse Proxy**: Requests to `/api/*` are proxied to the API container using its Docker service name as the hostname (`http://instance-api:3000/`). The trailing slash strips the `/api` prefix before forwarding to Fastify.

## 6. Database Naming and Seeding

To maintain strict parity across environments (development, staging, and production), the database connection and seeding architecture are decoupled from the infrastructure layer and handled directly at the application layer.

### 6.1. Database Naming Standardization

The system enforces `elodb` as the canonical database name across all environments. The database connection plugin parses the configured `MONGO_URI` connection string and overrides the database path to `/elodb`. This prevents accidental connection to default databases (such as `test` on MongoDB Atlas) and guarantees that development, staging, and production instances use the correct data store name.

### 6.2. Natively Integrated Seeding (`SeedPlugin`)

Database seeding is integrated into the API application lifecycle via a dedicated Fastify plugin (`seedPlugin.ts`). It triggers during the `onReady` hook of the Fastify server. Because it runs at the application level, it executes automatically in all environments (development, staging, and production).

### 6.3. Idempotent Upsert Strategy

Seeding operations must be completely safe to execute repeatedly without duplicating or corrupting data. The administrative user seeding utilizes an atomic Mongoose `findOneAndUpdate` operation with the following settings:

- **Upsert Filter**: Checks for the existence of a user with the `admin` role.
- **Set On Insert (`$setOnInsert`)**: Writes user fields (email, username, password, icon, role) only if no matching document exists.
- **Idempotency**: If an admin user is already present, the query resolves as a no-op, preventing existing credentials from being overwritten or duplicated on subsequent server restarts.

### 6.4. Infrastructure Separation of Concerns (`db-init`)

In local development, the container `db-init` handles replica set initialization only. It awaits MongoDB readiness and executes a dedicated script (`init-db.sh` inside the `init-scripts/` directory) to configure the single-node replica set (`rs0`) required for transaction support. It does not perform database seeding. All database record seeding responsibilities belong entirely to the API container.

## 7. Standard Directory Structure

Below is the gold-standard blueprint directory structure required for any containerized tooling or DevOps system context workspace within the repository:

```plaintext
<tool-or-workspace-name>/           # Root directory of the containerized tool or system context
├── infrastructure/                # Orchestration & Deployment Layer
│   ├── docker/
│   │   ├── compose.yaml           # Central Multi-Service Compose Orchestrator
│   │   ├── compose.override.yaml  # Developer local overrides configuration
│   │   ├── compose.prod.yaml      # Production & Staging overrides configuration
│   │   ├── .env.dev               # Centralized dev variables overrides
│   │   ├── .env.prod              # Centralized prod variables overrides
│   │   ├── .env.staging           # Centralized staging variables overrides
│   │   └── scripts/               # HOST-LEVEL SCRIPTS (Run on host VM / local machine)
│   │       ├── setup.sh           # Checks docker, prepares folders, sets sysctl VM parameters
│   │       ├── deploy.sh          # Handles image pulls, builds, container restarts, and pruning
│   │       ├── backup.sh          # Automates database dumps and volume cleanups
│   │       └── utils/
│   │           └── healthcheck.sh # Pings HTTP endpoints to verify correct startup
│   └── logs/                      # Isolated container execution and audit logs
├── scripts/                       # Repo-wide automation or helper TS/JS scripts
└── services/                      # Pluggable, isolated background runtime environments
    ├── primary-service/           # Primary daemon module (e.g. CLI or API engine)
    │   ├── Dockerfile             # Multi-stage optimized Docker build specification
    │   └── src/                   # Service-specific logic and application source code
    └── db-service/                # Stateful database helper (e.g. MongoDB)
        ├── Dockerfile
        ├── init-scripts/          # INITIALIZATION SCRIPTS (Executed once on bootstrap)
        │   └── init-db.sh         # Runs replica set initiation (e.g. rs.initiate)
        └── scripts/               # CONTAINER SCRIPTS (Copied and run inside container daemon)
            └── entrypoint.sh      # Intercepts daemon launch to generate keyfiles or configs
```

## 8. Gold-Standard Docker Compose Design

To orchestrate these modular workspaces without introducing global pollution or hard resource bounds, we employ a decoupled, environment-segregated, and non-blocking Docker Compose blueprint.

### 8.1. Gold-Standard `compose.yaml` Blueprint

```yaml
name: elo-tool-workspace # Named Docker Compose namespace to prevent container name clashes

services:
  # 1. Primary Service Daemon
  primary-service:
    build:
      context: ../../services/primary-service # Pluggable service build context (isolation)
      dockerfile: Dockerfile
    container_name: elo-primary-service-daemon # Strict standard container naming
    volumes:
      - ../../../../:/workspace # Volume mount to bind the host monorepo root
    env_file:
      - ../../services/primary-service/.env.service # Inject isolated, local service environment defaults
    environment:
      - API_TOKEN=${API_TOKEN} # Explicitly inject override secrets from the orchestrator .env
      - ENVIRONMENT=${ENVIRONMENT:-development}
    working_dir: /workspace # Consistent standard workspace path
    tty: true # Enable interactive tty (terminal) capability
    stdin_open: true # Keep stdin open for background container shell access
    command: ['tail', '-f', '/dev/null'] # Keep-alive idle loop to support 'docker exec' interaction

  # 2. Secondary Helper Daemon
  secondary-service:
    build:
      context: ../../services/secondary-service
      dockerfile: Dockerfile
    container_name: elo-secondary-service-daemon
    volumes:
      - ../../../../:/workspace
    env_file:
      - ../../services/secondary-service/.env.service
    environment:
      - HELPER_SECRET=${HELPER_SECRET}
    working_dir: /workspace
    tty: true
    stdin_open: true
    command: ['tail', '-f', '/dev/null']
```

### 8.2. Architectural Breakdown & Logic

#### 8.2.1. Namespace Isolation (`name`)

Each workspace must define a specific namespace using the `name` root property. This isolates container stacks, networks, and build caches, preventing host-wide name collisions across different contexts on the same Docker daemon.

#### 8.2.2. Scope-Bound Build Contexts (`build.context`)

The build context must point directly to the individual service directory under `services/`, rather than the monorepo root. This isolates the Docker build daemon's environment, avoiding sending the entire repository as build context. It drastically reduces RAM consumption, speeds up build caching, and enforces hermetic image compiles.

#### 8.2.3. Shared Workspace Volume Binding

Tooling and automation containers require direct access to the files in the repository. We mount the root directory to `/workspace` (using `- ../../../../:/workspace`) and establish it as the `working_dir`. This allows containerized binaries (such as linters, CLI tools, or parsers) to run and output changes directly on the host files while running in their isolated sandboxed environment.

#### 8.2.4. Environment Interface Segregation (`env_file` vs. `environment`)

To enforce the **Interface Segregation Principle**:

- Default, non-sensitive service configurations are loaded locally using `env_file` referencing service-specific defaults (e.g., `.env.service`).
- Deployment-wide secrets or external overrides are injected explicitly using the `environment` block (e.g., `${API_TOKEN}`), drawing values from the orchestration `.env` file placed under `infrastructure/docker/.env`.
- Containers never receive a monolithic global environment; they are bound strictly to their required inputs.

#### 8.2.5. Interactive Daemon Pattern (`tty` & Keep-Alive)

Many tooling and scripts do not run persistent web listening processes, which would cause the container to exit immediately after launching.

- By configuring `tty: true` and `stdin_open: true`, we enable interactive terminal capabilities.
- Setting `command: ['tail', '-f', '/dev/null']` starts a low-overhead, non-blocking background keep-alive loop.
- This creates an idle background daemon that stays active without consuming CPU, allowing developers or CI/CD pipelines to run commands inside the container on demand using:
  ```bash
  docker exec -it <container_name> <command>
  ```

## 9. Advanced Multi-Service Topologies

When building complex infrastructure workspaces within the monorepo, services are rarely isolated, single-daemon environments. Depending on the system's requirements, we classify multi-service layouts into two standard gold-standard topology patterns, both organized under the standard `services/` directory namespace to satisfy the Single Responsibility (SRP) and Open/Closed (OCP) principles.

---

### Pattern A: Central Hub with Pluggable Sub-Services (The Gateway Pattern)

This pattern is required for orchestrators, API gateways, or integration proxies (inspired by `tools/mcp/`) where a single core routing container mediates communication between multiple downstream client adapters, plugins, or services.

#### 9.1. Directory Blueprint (Pattern A)

```plaintext
<gateway-workspace>/
├── infrastructure/                 # Orchestration & Deployment Layer
│   └── docker/
│       ├── compose.yaml           # Central Multi-Service Compose Orchestrator
│       ├── .env                   # Deployment-wide variable overrides
│       └── .env.example           # Blueprint for deployment variables
├── scripts/                        # Host-Executed Automation and lifecycle TS/JS scripts
└── services/                       # Pluggable, isolated background runtime environments
    ├── gateway/                    # Primary routing/gateway daemon (e.g. CLI or API engine)
    │   ├── Dockerfile              # Multi-stage optimized Docker build specification
    │   ├── entrypoint.sh           # Unified entrypoint wrapper enforcing signaling/logging
    │   ├── .env.service.example    # Isolated, minimal local environment default blueprint
    │   └── src/                    # Central broker / reverse proxy server logic
    ├── service-adapter-a/          # Pluggable helper or specialized daemon module A (e.g., GitHub Adapter)
    │   ├── Dockerfile
    │   ├── entrypoint.sh
    │   ├── .env.service.example
    │   └── src/
    └── service-adapter-b/          # Pluggable helper or specialized daemon module B (e.g., Browser Adapter)
        ├── Dockerfile
        ├── entrypoint.sh
        ├── .env.service.example
        └── src/
```

#### 9.2. Architectural Design & SOLID Alignment (Pattern A)

- **Single Responsibility Principle (SRP)**: The central `gateway/` is exclusively responsible for traffic routing, authorization, and network dispatching. Each adapter directory under `services/` is strictly dedicated to its own standalone functional domain (such as file system access, browser automation, or third-party API integration).
- **Open/Closed Principle (OCP)**: Adding a new connector or adapter does not require modifying the gateway container's codebase. A new folder is simply introduced under `services/`, and its service entry is appended to `compose.yaml`.
- **Interface Segregation Principle (ISP)**: Each adapter has its own isolated `.env.service` configuration file and has no access to credentials of other adapters.

#### 9.3. Gold-Standard `compose.yaml` (Pattern A)

```yaml
name: elo-mcp-workspace # Named Docker Compose namespace

networks:
  mcp-net:
    name: elo-mcp-network
    driver: bridge

services:
  # 1. Central Routing Gateway
  gateway:
    build:
      context: ../../services/gateway
      dockerfile: Dockerfile
    container_name: elo-mcp-gateway
    restart: always
    environment:
      - PORT=3005
      - ENVIRONMENT=${ENVIRONMENT:-development}
    ports:
      - '3005:3005'
    networks:
      mcp-net:
        aliases:
          - mcp.gateway.internal
    depends_on:
      - adapter-github
      - adapter-browser

  # 2. Pluggable Adapter A (GitHub)
  adapter-github:
    build:
      context: ../../services/service-adapter-a
      dockerfile: Dockerfile
    container_name: elo-mcp-adapter-github
    restart: always
    env_file:
      - ../../services/service-adapter-a/.env.service
    environment:
      - GITHUB_TOKEN=${GITHUB_TOKEN} # Explicit injection of orchestration secrets
    networks:
      - mcp-net

  # 3. Pluggable Adapter B (Browser)
  adapter-browser:
    build:
      context: ../../services/service-adapter-b
      dockerfile: Dockerfile
    container_name: elo-mcp-adapter-browser
    restart: always
    env_file:
      - ../../services/service-adapter-b/.env.service
    shm_size: '2gb' # Prevent Chrome from crashing on complex pages
    networks:
      - mcp-net
```

### Pattern B: Equal-Peer Cooperating Services (The Distributed Stack Pattern)

This pattern is required for complex platforms (inspired by `studio/penpot/` or complete application stacks) where multiple specialized containers (frontend proxy, backend API core, in-memory caches, databases) must cooperate as equal peers without a single central routing hub.

#### 9.4. Directory Blueprint (Pattern B)

```plaintext
<distributed-workspace>/
├── config/                         # Unified configuration store for all peer services
│   ├── .env.frontend               # Specific environment keys for the UI container
│   ├── .env.backend                # Database and logic keys for the API container
│   └── .env.cache                  # Memory and cache boundaries
├── infrastructure/                 # Orchestration & Deployment Layer
│   └── docker/
│       ├── compose.yaml           # Central Compose Orchestrator
│       ├── .env                   # Shared deployment secrets
│       └── .env.example
├── scripts/                        # Workspace-wide database migrations or lifecycle scripts
└── services/                       # Fully modular, isolated peer services
    ├── frontend/                   # Equal-peer frontend engine (e.g. Nginx proxying static files)
    │   ├── Dockerfile
    │   ├── nginx.conf              # SPA route fallback and asset header configurations
    │   ├── entrypoint.sh
    │   └── src/
    ├── backend/                    # Equal-peer core API container
    │   ├── Dockerfile
    │   ├── entrypoint.sh
    │   └── src/                    # Controllers, business models, and service classes
    └── cache/                      # Peer service for high-performance memory store (e.g., Valkey or Redis)
        ├── Dockerfile
        └── valkey.conf
```

#### 9.5. Architectural Design & SOLID Alignment (Pattern B)

- **Interface Segregation Principle (ISP)**: No peer container should load environment variables or secrets it does not use. Environment config is separated into local files within the `config/` folder. The `compose.yaml` file then injects only the relevant files to each respective service container via the `env_file` block.
- **Liskov Substitution Principle (LSP) & Synchronization**: Containers must declare clean, uniform healthcheck contracts. Cooperative dependencies use the `depends_on` structure with a strict `service_healthy` condition:
  ```yaml
  backend:
    depends_on:
      cache:
        condition: service_healthy
  ```
  This ensures that downstream containers never start executing before their upstream peers have fully initialized and entered a ready state, maintaining stack reliability.
- **Strict Private Networking**: Peer services reside within a named, isolated virtual bridge network defined in `compose.yaml`. Only edge-facing entry points (such as the frontend service) are bound to external host ports, while stateful services (databases, caches) are hidden from the host network entirely.

#### 9.6. Gold-Standard `compose.yaml` (Pattern B)

```yaml
name: elo-distributed-workspace

networks:
  stack-net:
    name: elo-stack-network
    driver: bridge

services:
  # 1. Frontend Proxy (Edge Node)
  frontend:
    build:
      context: ../../services/frontend
      dockerfile: Dockerfile
    container_name: elo-stack-frontend
    restart: always
    ports:
      - '9005:8080' # Binds external host port to container edge
    env_file:
      - ../../config/.env.frontend
    depends_on:
      - backend
    networks:
      - stack-net

  # 2. Core API Backend
  backend:
    build:
      context: ../../services/backend
      dockerfile: Dockerfile
    container_name: elo-stack-backend
    restart: always
    env_file:
      - ../../config/.env.backend
    depends_on:
      cache:
        condition: service_healthy # Explicit state synchronization
    networks:
      - stack-net

  # 3. High-Performance Valkey Cache
  cache:
    build:
      context: ../../services/cache
      dockerfile: Dockerfile
    container_name: elo-stack-cache
    restart: always
    volumes:
      - stack_valkey_data:/data
    healthcheck:
      test: ['CMD-SHELL', 'valkey-cli ping | grep PONG']
      interval: 5s
      timeout: 5s
      retries: 5
    networks:
      - stack-net

volumes:
  stack_valkey_data:
```
