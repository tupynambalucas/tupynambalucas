# Padrões de Infraestrutura

Para manter a infraestrutura do **Elo Orgânico**, as ferramentas de DevOps e os ambientes de execução em segundo plano robustos, mantíveis e altamente modulares, este documento unifica nossos padrões de infraestrutura self-hosted, convenções de diretórios e topologias Docker multi-serviços.

---

## 1. Estratégia do Docker Compose (Desenvolvimento vs. Produção)

Projetado para **Excelência Self-Hosted** na Hetzner Cloud.

Cada contexto delimitado (ex: `instance/`, `portal/`) mantém uma configuração Compose modular utilizando herança/sobrescrita (overrides) de ambiente:

| Arquivo                 | Ambiente                       | Serviços                                   | Banco de Dados         |
| :---------------------- | :----------------------------- | :----------------------------------------- | :--------------------- |
| `compose.yaml`          | Configuração Base              | API + Web + Redis + MongoDB + Init         | -                      |
| `compose.override.yaml` | Desenvolvimento (Local)        | MongoDB Replica Set + Redis (Build/Portas) | Container Docker local |
| `compose.prod.yaml`     | Produção & Homologação/Staging | API + Web + Redis (Build/Limites/Réplicas) | MongoDB Atlas (nuvem)  |

Para instruções sobre como iniciar esses ambientes, leia a [Referência de Comandos](../commands-reference.mdx).

O `compose.prod.yaml` não possui serviço MongoDB por design. Em produção e homologação/staging, a API se conecta a um cluster MongoDB Atlas hospedado na nuvem via URI injetada a partir do arquivo `.env.prod` ou `.env.staging` específico do ambiente.

### 1.1. Convenções de Nomenclatura de Projetos e Containers

Para evitar colisões de nomes em hosts que executam múltiplas stacks ou ambientes simultaneamente (como homologação/staging e produção no mesmo servidor), as seguintes convenções são aplicadas:

- **Nomes de Projetos**: Os arquivos de compose de desenvolvimento definem `name: elo-[context]-dev` (ex: `elo-instance-dev`), enquanto os arquivos de compose de produção/staging definem `name: elo-[context]-prod` (ex: `elo-instance-prod`).
- **Ambientes de Desenvolvimento**: Os nomes dos containers possuem um sufixo explícito `-dev` (ex: `elo-instance-redis-dev`, `elo-instance-db-dev`).
- **Homologação vs. Produção**: Os containers de produção/staging anexam dinamicamente o sufixo do nome do ambiente usando a variável `APP_ENV` (ex: `container_name: elo-instance-redis-${APP_ENV:-prod}`). Isso isola os containers no nível do daemon do host Docker, permitindo que os containers de staging e produção coexistam na mesma máquina virtual sem colisões.

### 1.2. Por que Produção e Homologação/Staging Compartilham um Único Arquivo Compose

A decisão de usar um único `compose.prod.yaml` para produção e staging é uma escolha arquitetural deliberada, não um atalho de simplificação. A justificativa fundamenta-se no **Princípio de Responsabilidade Única (SRP)**:

Um arquivo Compose deve ter exatamente uma razão para mudar: uma **mudança na topologia da stack** (novos serviços, configuração de rede diferente, intervalos de healthcheck alterados, containers adicionais). Nem a produção nem o staging requerem uma topologia de serviço diferente — ambos executam o conjunto idêntico de serviços (API, Web/Nginx, Redis) conectados à mesma rede e com a mesma estratégia de healthcheck.

As únicas diferenças entre os dois ambientes são **valores de configuração** (URI do cluster MongoDB Atlas, chaves do Cloudflare Turnstile, segredos de API, chaves de tag de imagem). Esta é uma preocupação de configuração, não de topologia, e é de responsabilidade exclusiva dos arquivos `.env` específicos de cada ambiente (`.env.prod`, `.env.staging`).

Criar um `compose.staging.yaml` separado como uma cópia estrutural de `compose.prod.yaml` violaria o SRP ao introduzir dois arquivos com o mesmo motivo para mudar. Qualquer alteração futura de topologia (adicionar um container de monitoramento, atualizar intervalos de healthcheck) exigiria edições idênticas em ambos os arquivos, gerando risco de divergência. O nível correto de segregação é a flag `--env-file`, que é o mecanismo recomendado pelo Docker para injeção de configurações específicas de ambiente.

## 2. Arquitetura dos Dockerfiles

Cada aplicação possui seu próprio `Dockerfile` localizado dentro do seu respectivo diretório, seguindo os princípios de **Responsabilidade Única** e de **Contexto Delimitado**:

```plaintext
instance/services/api/Dockerfile    # Multi-stage: turbo prune → build → pnpm deploy → runner
instance/services/web/Dockerfile    # Multi-stage: turbo prune → vite build → nginx:alpine
portal/apps/api/Dockerfile      # Mesmo padrão que instance/api
portal/apps/web/Dockerfile      # Mesmo padrão que instance/web
```

Serviços de banco de dados e cache (MongoDB, Redis) possuem seus Dockerfiles e arquivos de ambiente locais encapsulados dentro de `instance/services/mongodb/` e `instance/services/redis/` respectivamente, mantendo-os completamente autocontidos.

Os Dockerfiles de API utilizam a **poda do Turborepo** (`turbo prune`) e o **`pnpm deploy`** para produzir imagens mínimas — sem código-fonte, sem devDependencies e sem symlinks do monorepo em produção.

Os Dockerfiles Web utilizam uma compilação de dois estágios (two-stage build): um construtor Node.js compila o SPA React via Vite (com variáveis `VITE_*` injetadas como `ARG` do Docker) e o estágio final serve os arquivos estáticos a partir do `nginx:alpine` (imagem de aproximadamente 25MB).

## 3. Estratégia de Variáveis de Ambiente

As variáveis de ambiente seguem os princípios de **Contexto Delimitado** e **Responsabilidade Única**. Para eliminar a redundância e evitar a divergência de configurações, todos os arquivos de variáveis de ambiente são centralizados sob o escopo do orquestrador (`[context]/infrastructure/docker/`). Os serviços são mantidos completamente sem estado (stateless) e carregam suas configurações por meio de injeção no nível de processo:

| Padrão de Arquivo                                | Dono / Alvo          | Propósito / Escopo                                                                                            | Git           |
| :----------------------------------------------- | :------------------- | :------------------------------------------------------------------------------------------------------------ | :------------ |
| `[context]/infrastructure/docker/.env.dev`       | Stack de Dev Local   | Variáveis centralizadas para desenvolvimento (configuração da API no host, credenciais locais do DB/Redis)    | ignorado      |
| `[context]/infrastructure/docker/.env.prod`      | Stack de Produção    | Segredos de produção centralizados, URIs do MongoDB Atlas, credenciais de produção do Turnstile               | ignorado      |
| `[context]/infrastructure/docker/.env.staging`   | Stack de Homologação | Configurações de staging centralizadas, URIs do MongoDB Atlas de staging, credenciais de staging do Turnstile | ignorado      |
| `[context]/infrastructure/docker/.env.*.example` | Repositório          | Modelos de configuração de onboarding para os escopos de implantação de dev, prod e staging                   | **rastreado** |

_Nota: Na tabela acima, `[context]` representa o diretório `instance` ou `portal`._

As variáveis `VITE_*` (como `VITE_TURNSTILE_SITE_KEY`) são apenas para o **tempo de build (compilação)** — o Vite as grava estaticamente no bundle JavaScript durante o `vite build`. Em builds Docker, elas são transmitidas via `ARG` → `ENV` antes do início da etapa de compilação. No desenvolvimento local, o Vite carrega as variáveis diretamente do diretório centralizado `[context]/infrastructure/docker/.env.dev` por meio de sua configuração `envDir`.

## 4. Rede & Latência

Todos os componentes em uma stack de implantação residem em uma rede privada bridge nomeada do Docker (ex: `elo_instance_network`) para latência sub-milissegundo entre a API e o Redis. O container Nginx é o único serviço com um mapeamento de porta externa; a API só é alcançável através do proxy reverso do Nginx na rede interna.

---

## 5. Configuração do Nginx

Os containers Web (Nginx) lidam com duas preocupações principais:

1. **Roteamento SPA**: Todos os caminhos desconhecidos retornam para o `index.html` via `try_files $uri $uri/ /index.html`, permitindo que o React Router gerencie a navegação no lado do cliente sem gerar erros 404 ao atualizar a página.
2. **Proxy Reverso de API**: As requisições para `/api/*` são direcionadas ao container da API usando o nome do seu serviço Docker como hostname (`http://instance-api:3000/`). A barra no final remove o prefixo `/api` antes de encaminhar a requisição para o Fastify.

## 6. Nomenclatura e Semeadura de Banco de Dados

Para manter paridade estrita entre os diferentes ambientes (desenvolvimento, staging e produção), a conexão de banco de dados e a arquitetura de semeadura (seeding) são desacopladas da camada de infraestrutura e gerenciadas diretamente na camada de aplicação.

### 6.1. Padronização do Nome do Banco de Dados

O sistema impõe `elodb` como o nome canônico do banco de dados em todos os ambientes. O plugin de conexão de banco de dados analisa a string de conexão `MONGO_URI` configurada e substitui o caminho do banco de dados por `/elodb`. Isso evita conexões acidentais a bancos de dados padrão (como `test` no MongoDB Atlas) e garante que as instâncias de desenvolvimento, staging e produção utilizem o nome correto do data store.

### 6.2. Semeadura Integrada Nativamente (`SeedPlugin`)

A semeadura do banco de dados é integrada ao ciclo de vida da aplicação API através de um plugin dedicado do Fastify (`seedPlugin.ts`). Ela é disparada durante o hook `onReady` do servidor Fastify. Como é executada no nível da aplicação, ocorre automaticamente em todos os ambientes (desenvolvimento, staging e produção).

### 6.3. Estratégia de Upsert Idempotente

As operações de semeadura devem ser totalmente seguras para execução repetida sem duplicar ou corromper dados. A semeadura do usuário administrativo utiliza uma operação atômica do Mongoose `findOneAndUpdate` com as seguintes configurações:

- **Filtro de Upsert**: Verifica a existência de um usuário com o papel de `admin`.
- **Set On Insert (`$setOnInsert`)**: Grava os campos do usuário (email, username, password, icon, role) apenas se nenhum documento correspondente for encontrado.
- **Idempotência**: Se um usuário administrador já estiver presente, a consulta é resolvida como um no-op (sem operação), evitando que credenciais existentes sejam sobrescritas ou duplicadas em reinicializações subsequentes do servidor.

### 6.4. Separação de Preocupações na Infraestrutura (`db-init`)

No desenvolvimento local, o container `db-init` lida exclusivamente com a inicialização do replica set. Ele aguarda a prontidão do MongoDB e executa um script dedicado (`init-db.sh` dentro do diretório `init-scripts/`) para configurar o replica set de nó único (`rs0`) necessário para suporte a transações. Ele não realiza semeadura de banco de dados. Todas as responsabilidades de semeadura de registros de banco de dados pertencem inteiramente ao container da API.

## 7. Estrutura de Diretórios Padrão

Abaixo está a estrutura de diretórios padrão ouro exigida para qualquer workspace de ferramenta conteinerizada ou contexto de sistema de DevOps no repositório:

```plaintext
<nome-da-ferramenta-ou-workspace>/ # Diretório raiz do contexto de ferramenta conteinerizada ou sistema
├── infrastructure/                # Camada de Orquestração & Implantação (Deployment)
│   ├── docker/
│   │   ├── compose.yaml           # Orquestrador Central Multi-Serviço (Compose)
│   │   ├── compose.override.yaml  # Configurações de sobrescrita local para desenvolvedor
│   │   ├── compose.prod.yaml      # Configurações de sobrescrita para produção e staging
│   │   ├── .env.dev               # Sobrescritas de variáveis centralizadas de desenvolvimento
│   │   ├── .env.prod              # Sobrescritas de variáveis centralizadas de produção
│   │   ├── .env.staging           # Sobrescritas de variáveis centralizadas de staging
│   │   └── scripts/               # SCRIPTS DE HOST (Executados na máquina virtual / host local)
│   │       ├── setup.sh           # Valida o docker, prepara pastas do host e limites de kernel
│   │       ├── deploy.sh          # Realiza o pull, build, reinicialização e limpeza de containers
│   │       ├── backup.sh          # Automatiza o dump do banco de dados e arquivos locais
│   │       └── utils/
│   │           └── healthcheck.sh # Pings de verificação de disponibilidade HTTP pós-deploy
│   └── logs/                      # Logs de execução e auditoria de containers isolados
├── scripts/                       # Scripts de automação ou utilitários gerais do repositório
└── services/                      # Ambientes de execução em segundo plano isolados e plugáveis
    ├── primary-service/           # Módulo daemon plugável (ex: api)
    │   ├── Dockerfile             # Especificação otimizada de build Docker multi-stage
    │   └── src/                   # Lógica específica do serviço e código da aplicação
    └── db-service/                # Módulo de banco de dados persistente (ex: mongodb)
        ├── Dockerfile
        ├── init-scripts/          # SCRIPTS DE INICIALIZAÇÃO (Executados uma vez no bootstrap)
        │   └── init-db.sh         # Inicializa o replica set do banco de dados (ex: rs.initiate)
        └── scripts/               # SCRIPTS DE CONTAINER (Copiados e executados dentro do container)
            └── entrypoint.sh      # Intercepta a inicialização do container para gerar arquivos de chave
```

## 8. Design de Docker Compose Padrão Ouro

Para orquestrar esses workspaces modulares sem introduzir poluição global ou limites rígidos de recursos, empregamos um modelo do Docker Compose desacoplado, segregado por ambiente e não bloqueante.

### 8.1. Modelo do `compose.yaml` Padrão Ouro

```yaml
name: elo-tool-workspace # Named Docker Compose namespace to prevent container name clashes

services:
  # 1. Primary Service Daemon
  primary-service:
    build:
      context: ../../services/primary-service # Pluggable service build context (isolation)
      dockerfile: Dockerfile
    container_name: elo-primary-service-daemon # Strict standard container naming
    volumes:
      - ../../../../:/workspace # Volume mount to bind the host monorepo root
    env_file:
      - ../../services/primary-service/.env.service # Inject isolated, local service environment defaults
    environment:
      - API_TOKEN=${API_TOKEN} # Explicitly inject override secrets from the orchestrator .env
      - ENVIRONMENT=${ENVIRONMENT:-development}
    working_dir: /workspace # Consistent standard workspace path
    tty: true # Enable interactive tty (terminal) capability
    stdin_open: true # Keep stdin open for background container shell access
    command: ['tail', '-f', '/dev/null'] # Keep-alive idle loop to support 'docker exec' interaction

  # 2. Secondary Helper Daemon
  secondary-service:
    build:
      context: ../../services/secondary-service
      dockerfile: Dockerfile
    container_name: elo-secondary-service-daemon
    volumes:
      - ../../../../:/workspace
    env_file:
      - ../../services/secondary-service/.env.service
    environment:
      - HELPER_SECRET=${HELPER_SECRET}
    working_dir: /workspace
    tty: true
    stdin_open: true
    command: ['tail', '-f', '/dev/null']
```

### 8.2. Análise Arquitetural & Lógica

#### 8.2.1. Isolamento de Namespace (`name`)

Cada workspace deve definir um namespace específico usando a propriedade raiz `name`. Isso isola as pilhas (stacks) de containers, redes e caches de build, evitando colisões de nomes em todo o host entre diferentes contextos no mesmo daemon do Docker.

#### 8.2.2. Contextos de Build Limitados ao Escopo (`build.context`)

O contexto de build deve apontar diretamente para o diretório de serviço individual sob `services/`, e não para a raiz do monorepo. Isso isola o ambiente do daemon de build do Docker, evitando enviar todo o repositório como contexto de compilação. Isso reduz drasticamente o consumo de RAM, acelera o cache do build e impõe compilações de imagem herméticas.

#### 8.2.3. Associação de Volume do Workspace Compartilhado (Volume Binding)

As ferramentas e containers de automação exigem acesso direto aos arquivos no repositório. Mapeamos o diretório raiz para `/workspace` (usando `- ../../../../:/workspace`) e o estabelecemos como o `working_dir`. Isso permite que binários conteinerizados (como linters, ferramentas de CLI ou parsers) executem e salvem alterações diretamente nos arquivos do host enquanto rodam em seu ambiente isolado (sandbox).

#### 8.2.4. Segregação de Interfaces de Ambiente (`env_file` vs. `environment`)

Para impor o **Princípio da Segregação de Interface**:

- As configurações padrão e não sensíveis de serviço são carregadas localmente usando `env_file` referenciando os padrões específicos do serviço (por exemplo, `.env.service`).
- Segredos de todo o deployment ou sobrescritas externas são injetados explicitamente usando o bloco `environment` (por exemplo, `${API_TOKEN}`), extraindo valores do arquivo `.env` de orquestração posicionado sob `infrastructure/docker/.env`.
- Os containers nunca recebem um ambiente global monolítico; eles são vinculados estritamente às entradas de que precisam.

#### 8.2.5. Padrão de Daemon Interativo (`tty` & Keep-Alive)

Muitas ferramentas e scripts não executam processos de escuta na web persistentes, o que faria com que o container fosse encerrado imediatamente após o lançamento.

- Ao configurar `tty: true` e `stdin_open: true`, ativamos recursos interativos do terminal.
- Definir `command: ['tail', '-f', '/dev/null']` inicia um loop keep-alive de segundo plano leve e não bloqueante.
- Isso cria um daemon inativo em segundo plano que permanece ativo sem consumir CPU, permitindo que os desenvolvedores ou pipelines de CI/CD executem comandos dentro do container sob demanda usando:
  ```bash
  docker exec -it <nome_do_container> <comando>
  ```

## 9. Topologias Avançadas de Multi-Serviços

Ao construir espaços de infraestrutura complexos dentro do monorepo, os serviços raramente são ambientes isolados de um único daemon. Dependendo dos requisitos do sistema, classificamos os layouts de multi-serviços em dois padrões padrão ouro de topologia, ambos organizados sob o namespace de diretório padrão `services/` para satisfazer os princípios de Responsabilidade Única (SRP) e Aberto/Fechado (OCP).

---

### Padrão A: Hub Central com Sub-Serviços Plugáveis (O Padrão Gateway)

Este padrão é exigido para orquestradores, gateways de API ou proxies de integração (inspirado no `tools/mcp/`), onde um único container de roteamento principal media a comunicação entre múltiplos adaptadores de clientes downstream, plugins ou serviços.

#### 9.1. Modelo de Diretórios (Padrão A)

```plaintext
<gateway-workspace>/
├── infrastructure/                 # Camada de Orquestração & Implantação (Deployment)
│   └── docker/
│       ├── compose.yaml           # Orquestrador Central Multi-Serviço (Compose)
│       ├── .env                   # Sobrescritas de variáveis para todo o deployment
│       └── .env.example           # Modelo para variáveis de deployment
├── scripts/                       # Automação Executada no Host e scripts TS/JS de ciclo de vida
└── services/                      # Ambientes de execução em segundo plano isolados e plugáveis
    ├── gateway/                    # Daemon de roteamento/gateway primário (ex: engine de CLI ou API)
    │   ├── Dockerfile              # Especificação otimizada de build Docker multi-stage
    │   ├── entrypoint.sh           # Wrapper de entrypoint unificado que impõe sinais/logs
    │   ├── .env.service.example    # Modelo isolado e minimalista de variáveis locais padrão
    │   └── src/                    # Lógica do servidor de broker central / proxy reverso
    ├── service-adapter-a/          # Módulo daemon helper ou especializado plugável A (ex: Adaptador GitHub)
    │   ├── Dockerfile
    │   ├── entrypoint.sh
    │   ├── .env.service.example
    │   └── src/
    └── service-adapter-b/          # Módulo daemon helper ou especializado plugável B (ex: Adaptador de Navegador)
        ├── Dockerfile
        ├── entrypoint.sh
        ├── .env.service.example
        └── src/
```

#### 9.2. Design Arquitetural & Alinhamento SOLID (Padrão A)

- **Princípio da Responsabilidade Única (SRP)**: O `gateway/` central é exclusivamente responsável pelo roteamento de tráfego, autorização e despacho de rede. Cada diretório de adaptador sob `services/` é estritamente dedicado ao seu próprio domínio funcional independente (como acesso ao sistema de arquivos, automação de navegador ou integração de APIs de terceiros).
- **Princípio Aberto/Fechado (OCP)**: Adicionar um novo conector ou adaptador não exige a modificação do código-fonte do container do gateway. Uma nova pasta é simplesmente introduzida sob `services/` e sua entrada de serviço é anexada ao `compose.yaml`.
- **Princípio da Segregação de Interface (ISP)**: Cada adaptador tem seu próprio arquivo de configuração `.env.service` isolado e não tem acesso às credenciais de outros adaptadores.

#### 9.3. `compose.yaml` Padrão Ouro (Padrão A)

```yaml
name: elo-mcp-workspace # Namespace nomeado do Docker Compose

networks:
  mcp-net:
    name: elo-mcp-network
    driver: bridge

services:
  # 1. Gateway de Roteamento Central
  gateway:
    build:
      context: ../../services/gateway
      dockerfile: Dockerfile
    container_name: elo-mcp-gateway
    restart: always
    environment:
      - PORT=3005
      - ENVIRONMENT=${ENVIRONMENT:-development}
    ports:
      - '3005:3005'
    networks:
      mcp-net:
        aliases:
          - mcp.gateway.internal
    depends_on:
      - adapter-github
      - adapter-browser

  # 2. Adaptador Plugável A (GitHub)
  adapter-github:
    build:
      context: ../../services/service-adapter-a
      dockerfile: Dockerfile
    container_name: elo-mcp-adapter-github
    restart: always
    env_file:
      - ../../services/service-adapter-a/.env.service
    environment:
      - GITHUB_TOKEN=${GITHUB_TOKEN} # Injeção explícita de segredos da orquestração
    networks:
      - mcp-net

  # 3. Adaptador Plugável B (Browser)
  adapter-browser:
    build:
      context: ../../services/service-adapter-b
      dockerfile: Dockerfile
    container_name: elo-mcp-adapter-browser
    restart: always
    env_file:
      - ../../services/service-adapter-b/.env.service
    shm_size: '2gb' # Evita que o Chrome quebre em páginas complexas
    networks:
      - mcp-net
```

### Padrão B: Serviços Cooperativos de Pares Iguais (O Padrão Stack Distribuída)

Este padrão é exigido para plataformas complexas (inspirado no `studio/penpot/` ou stacks completas de aplicações) onde múltiplos containers especializados (proxy frontend, núcleo de API backend, caches em memória, bancos de dados) devem cooperar como pares iguais, sem um único hub de roteamento central.

#### 9.4. Modelo de Diretórios (Padrão B)

```plaintext
<distributed-workspace>/
├── config/                         # Armazenamento de configuração unificado para todos os serviços pares
│   ├── .env.frontend               # Chaves de ambiente específicas para o container de interface do usuário (UI)
│   ├── .env.backend                # Chaves de banco de dados e lógica para o container de API
│   └── .env.cache                  # Limites de memória e cache
├── infrastructure/                 # Camada de Orquestração & Implantação (Deployment)
│   └── docker/
│       ├── compose.yaml           # Orquestrador Central Multi-Serviço (Compose)
│       ├── .env                   # Segredos de implantação compartilhados
│       └── .env.example
├── scripts/                        # Migrações de banco de dados do workspace ou scripts de ciclo de vida
└── services/                       # Serviços pares totalmente modulares e isolados
    ├── frontend/                   # Engine frontend de par igual (ex: Nginx servindo arquivos estáticos)
    │   ├── Dockerfile
    │   ├── nginx.conf              # Configurações de rota fallback de SPA e cabeçalhos de ativos
    │   ├── entrypoint.sh
    │   └── src/
    ├── backend/                    # Container de API principal de par igual
    │   ├── Dockerfile
    │   ├── entrypoint.sh
    │   └── src/                    # Controllers, modelos de negócios e classes de serviço
    └── cache/                      # Serviço de par para cache em memória de alta performance (ex: Valkey ou Redis)
        ├── Dockerfile
        └── valkey.conf
```

#### 9.5. Design Arquitetural & Alinhamento SOLID (Padrão B)

- **Princípio da Segregação de Interface (ISP)**: Nenhum container par deve carregar variáveis de ambiente ou segredos que não utiliza. A configuração de ambiente é separada em arquivos locais dentro da pasta `config/`. O arquivo `compose.yaml` então injeta apenas os arquivos relevantes para cada respectivo container de serviço através do bloco `env_file`.
- **Princípio de Substituição de Liskov (LSP) & Sincronização**: Os containers devem declarar contratos de verificação de integridade (healthcheck) limpos e uniformes. Dependências cooperativas usam a estrutura `depends_on` com uma condição estrita de `service_healthy`:
  ```yaml
  backend:
    depends_on:
      cache:
        condition: service_healthy
  ```
  Isso garante que os containers downstream nunca comecem a ser executados antes que seus pares upstream tenham sido totalmente inicializados e entrado em um estado pronto, mantendo a confiabilidade da stack.
- **Rede Privada Estrita**: Os serviços de pares residem dentro de uma rede de ponte virtual (bridge) isolada e nomeada, definida no `compose.yaml`. Apenas os pontos de entrada voltados para o exterior (como o serviço de frontend) são vinculados a portas de host externas, enquanto serviços com estado (bancos de dados, caches) ficam inteiramente ocultos da rede do host.

#### 9.6. `compose.yaml` Padrão Ouro (Padrão B)

```yaml
name: elo-distributed-workspace

networks:
  stack-net:
    name: elo-stack-network
    driver: bridge

services:
  # 1. Proxy Frontend (Nó de Borda)
  frontend:
    build:
      context: ../../services/frontend
      dockerfile: Dockerfile
    container_name: elo-stack-frontend
    restart: always
    ports:
      - '9005:8080' # Vincula a porta externa do host à borda do container
    env_file:
      - ../../config/.env.frontend
    depends_on:
      - backend
    networks:
      - stack-net

  # 2. Backend de API Principal
  backend:
    build:
      context: ../../services/backend
      dockerfile: Dockerfile
    container_name: elo-stack-backend
    restart: always
    env_file:
      - ../../config/.env.backend
    depends_on:
      cache:
        condition: service_healthy # Sincronização explícita de estado
    networks:
      - stack-net

  # 3. Cache Valkey de Alta Performance
  cache:
    build:
      context: ../../services/cache
      dockerfile: Dockerfile
    container_name: elo-stack-cache
    restart: always
    volumes:
      - stack_valkey_data:/data
    healthcheck:
      test: ['CMD-SHELL', 'valkey-cli ping | grep PONG']
      interval: 5s
      timeout: 5s
      retries: 5
    networks:
      - stack-net

volumes:
  stack_valkey_data:
```
