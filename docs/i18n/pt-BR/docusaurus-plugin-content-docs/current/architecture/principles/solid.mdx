# Princípios SOLID

Os princípios de design SOLID orientam a estrutura orientada a objetos e funcional tanto do backend Fastify quanto do frontend React, garantindo sistemas altamente modulares, testáveis e robustos.

---

## 1. Princípio de Responsabilidade Única (SRP)

### Conceito & Visão Geral

Um módulo, classe ou função deve ter uma, e apenas uma, razão para mudar, o que significa que ele deve encapsular uma única tarefa bem definida. O SRP reduz o acoplamento e garante que uma alteração em um requisito do sistema afete apenas um único módulo isolado.

### Utilização Específica no Projeto

#### Backend (API Fastify)

Impomos uma arquitetura em camadas rígida:
`Controller` -> `Service` -> `Repository` -> `Model`

- **Controllers** apenas lidam com o roteamento bruto de requisições/respostas, análise de dados e validação do Zod.
- **Services** orquestram exclusivamente as regras de negócio, limites de transações e integrações.
- **Repositories** gerenciam consultas e mutações brutas de banco de dados.
- **Models** definem esquemas do Mongoose e estruturas de validação.

#### Frontend (Aplicação Web React)

Os componentes React são desacoplados do gerenciamento de estado e buscas assíncronas:

- **As stores do Zustand** são restritas a mutações de estado e disparadores de ações brutas.
- **Os hooks personalizados do React** (`src/domains/*/hooks`) lidam com seletores de estado e disparam ações de API, mantendo os componentes de UI do React estritamente focados na renderização.

## 2. Princípio Aberto/Fechado (OCP)

### Conceito & Visão Geral

Módulos de software devem ser abertos para extensão, mas fechados para modificação. Novos recursos ou comportamentos devem ser adicionados estendendo o código existente (ex: por meio de herança, composição ou plugins), não alterando a lógica central já verificada.

### Utilização Específica no Projeto

#### Backend (API Fastify)

O ecossistema de plugins e decoradores do Fastify é uma implementação ideal do OCP. Nós estendemos as capacidades centrais do Fastify (como autenticadores ou conexões de banco de dados) registrando plugins utilizando `fastify-plugin` sem alterar o bootstrapper principal do servidor:

```typescript
// Registrando um decorador de autenticação de forma segura
// instance/apps/api/src/plugins/auth.ts
import fp from 'fastify-plugin';

export default fp(async (fastify) => {
  fastify.decorate('authenticate', async (request, reply) => {
    // Lógica do autenticador aqui...
  });
});
```

#### Frontend (Aplicação Web React)

Aproveitamos amplamente o padrão de **Composição de Componentes do React** em vez de inflar componentes individuais com condicionais complexas. Elementos de UI genéricos (tabelas, diálogos, cards) aceitam layouts customizados via propriedades de filhos (children) ou funções de renderização tipadas, evitando lógicas condicionais complexas com if-else dentro do componente para lidar com variações de estilos ou conteúdos:

```tsx
// Aberto para extensão via composição, fechado para modificação
interface CardProps {
  title: string;
  children: React.ReactNode;
  actions?: React.ReactNode;
}

export const PremiumCard = ({ title, children, actions }: CardProps) => (
  <div className="card-container">
    <div className="card-header">
      <h3>{title}</h3>
      {actions !== undefined && <div className="card-actions">{actions}</div>}
    </div>
    <div className="card-body">{children}</div>
  </div>
);
```

## 3. Princípio de Substituição de Liskov (LSP)

### Conceito & Visão Geral

Subtipos devem ser completamente substituíveis por seus tipos base sem alterar a corretude ou comportamento da aplicação. Se um serviço depende de uma interface, qualquer classe que implemente essa interface deve ser intercambiável sem quebrar o código do cliente.

### Utilização Específica no Projeto

#### Adaptadores de Armazenamento de Arquivos

Todos os adaptadores externos, provedores de cache e modelos de banco de dados são projetados com base em interfaces rígidas e unificadas. Por exemplo, nossos gerenciadores de upload de arquivos implementam um contrato comum e tipado:

```typescript
export interface IFileStorageAdapter {
  uploadFile(file: Buffer, path: string): Promise<string>;
  deleteFile(path: string): Promise<void>;
}
```

No desenvolvimento local, injetamos um `LocalDiskStorageAdapter` que grava em `tmp/`. Em produção e homologação, injetamos um `CloudflareR2StorageAdapter` ou `S3StorageAdapter`. Ambos implementam `IFileStorageAdapter` perfeitamente. O serviço consumidor (ex: `ProductImageService`) permanece 100% agnóstico ao sistema subjacente, e podemos substituí-los de forma transparente sem quebrar a lógica de negócio do serviço.

## 4. Princípio de Segregação de Interface (ISP)

### Conceito & Visão Geral

Clientes não devem ser forçados a depender de interfaces ou métodos que não utilizam. Interfaces pequenas e específicas são sempre superiores a um contrato único e generalista de propósito geral.

### Utilização Específica no Projeto

#### Subcontratos Tipados

Evitamos expor grandes modelos monolíticos ou documentos de banco de dados diretamente para camadas que não precisam deles. Em vez disso, declaramos subinterfaces e esquemas do Zod específicos e reduzidos dentro da biblioteca `@elo-instance/core`:

```typescript
// Representação monolítica do banco de dados (Apenas interno)
export interface IUserDocument extends Document {
  id: string;
  email: string;
  passwordHash: string;
  username: string;
  role: 'admin' | 'user';
  createdAt: Date;
  updatedAt: Date;
}

// Contratos segregados para camadas downstream
export interface IUserPublicSession {
  id: string;
  email: string;
  role: 'admin' | 'user';
}

export interface ICreateUserDTO {
  email: string;
  username: string;
}
```

Os módulos consumidores, como os componentes de cabeçalho (header) do frontend, importam e dependem exclusivamente da pequena interface `IUserPublicSession` em vez do documento completo do banco de dados, garantindo que alterações nas regras de hash de senhas não exijam refatoração nos componentes de tela.

## 5. Princípio de Inversão de Dependência (DIP)

### Conceito & Visão Geral

Módulos de alto nível não devem depender diretamente de módulos de baixo nível; ambos devem depender de abstrações (ex: interfaces ou classes abstratas). Detalhes devem depender de abstrações, e não o contrário.

### Utilização Específica no Projeto

#### Backend (Injeção de Dependência)

Os serviços nunca instanciam repositórios, bancos de dados ou SDKs de terceiros diretamente dentro de seus construtores. Em vez disso, as dependências são passadas para os serviços por meio de injeção no construtor:

```typescript
import type { Model } from 'mongoose';
import type { IUser } from '@elo-instance/core';

export class AuthRepository {
  // Modelo injetado, desacoplado de instanciação concreta
  constructor(private readonly userModel: Model<IUser>) {}

  async findByEmail(email: string): Promise<IUser | null> {
    return this.userModel.findOne({ email }).exec();
  }
}
```

Esse desacoplamento nos permite injetar facilmente modelos mockados do Mongoose em nossos testes unitários, isolando completamente o banco de dados das nossas suítes de teste.

#### Frontend (Abstrações de Serviços)

Os componentes React não realizam chamadas diretas a endpoints do Axios ou acessam o `localStorage`. Em vez disso, interagem com hooks de domínio customizados e selecionam métodos das stores do Zustand. A implementação concreta do cliente HTTP e da persistência do navegador fica oculta por trás da abstração da camada de domínio, permitindo-nos alterar a biblioteca de rede (ex: substituir o Axios pelo Fetch do navegador) sem modificar nossos componentes de UI.
