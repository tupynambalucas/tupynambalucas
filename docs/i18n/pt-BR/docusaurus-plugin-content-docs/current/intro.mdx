# Projeto TupyDocs

Este projeto é uma plataforma de gestão especializada projetada para preencher a lacuna entre produtores orgânicos e consumidores conscientes através de **Ciclos de Compartilhamento Agendados**. Mais do que apenas uma ferramenta de e-commerce, é um sistema de gestão orientado pela comunidade que automatiza a complexa logística de colheitas sazonais e compras em grupo.

_"Tornar a comida local, orgânica e sustentável tão acessível e fácil de comprar quanto qualquer produto industrial comum, sem perder a conexão humana."_

## 1. Visão Geral do Produto e Proposta de Valor

Antes do Elo Orgânico, o processo de compartilhamento era dificultado por:

- **Caos Operacional:** Dependência de planilhas massivas e mensagens de WhatsApp fragmentadas.
- **Erros Manuais:** Equívocos humanos na consolidação de pedidos e rastreamento de pagamentos.
- **Atrito de Escalonamento:** Alta sobrecarga administrativa tornava impossível o crescimento da comunidade sem sobrecarga (burnout).
- **Latência de Pagamento:** Conciliação financeira atrasada devido à verificação manual de transferências bancárias.

Nós transformamos esse processo em um fluxo de trabalho profissional e automatizado:

| Para Consumidores                                                                        | Para Administradores/Fornecedores                                                                 |
| :--------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------ |
| **Conveniência:** Uma loja limpa e moderna (`@elo-instance/web`) para navegar e comprar. | **Automação:** Ingestão inteligente de listas de produtos em dados estruturados.                  |
| **Transparência:** Acesso em tempo real à disponibilidade sazonal e preços.              | **Eficiência:** Consolidação automatizada de pedidos e gestão de estoque via `@elo-instance/api`. |
| **Velocidade:** Pagamentos Pix instantâneos com conciliação bancária automática.         | **Controle:** Visibilidade total sobre ciclos, logística e saúde financeira.                      |

### Público-Alvo (Personas)

1. **O Coordenador de Comunidade (Admin):** Gerencia ciclos e fornecedores através das funcionalidades administrativas do `@elo-instance/web`.
2. **O Consumidor Consciente:** Compra produtos sazonais através da interface do `@elo-instance/web`.
3. **O Fornecedor:** Atua como a ponte entre os agricultores e a comunidade, fornecendo o catálogo sazonal.

## 2. Fluxograma Operacional

A plataforma Elo Orgânico orquestra o seguinte fluxo operacional para uma instância de comunidade:

1. **Configuração do Ciclo:** O administrador define e agenda janelas de compra (ciclos) através da interface administrativa do `@elo-instance/web`. `[ESTÁVEL]`
2. **Ingestão de Produtos:** Uma lista de produtos em texto puro é processada por um **Parser Inteligente** no frontend. `[WIP - Refinando Lógica]`
3. **Abertura de Compras:** O catálogo estruturado é habilitado automaticamente para os membros na loja `@elo-instance/web`. `[ESTÁVEL]`
4. **Execução de Pedidos:** Os membros montam cestas personalizadas e enviam seus pedidos. `[ESTÁVEL]`
5. **Processamento de Pagamento:** Integração direta com a API do **Banco EFI** para pagamentos via Pix. `[WIP - Fase de Especificação Técnica]`
6. **Confirmação:** O sistema realiza a conciliação bancária e confirma o pedido automaticamente. `[WIP]`

## 3. Estimativa de Custos e Estratégia de Hospedagem

A estratégia "Self-Hosted" proporciona uma relação custo-benefício altamente favorável e previsível. Propomos duas abordagens principais de hospedagem para a instância.

### Categoria 1: Infraestrutura e Hospedagem

**Opção A: Hetzner Cloud (VPS Fixo)**

- **Estratégia:** Servidor de Alta Performance (Linux VPS) sempre ligado.
- **Custo Mensal Estimado:** **€5.00 a €7.00** (Fixo).
- **Banco de Dados & Cache:** Recursos da própria VPS (MongoDB/Redis) sem custo extra.

**Opção B: Google Cloud Run (Serverless)**

- **Estratégia:** Containers Serverless, ligados e desligados sob demanda.
- **Custo Mensal Estimado:** **Variável** (Frequentemente free tier ou centavos se ocioso).
- **Banco de Dados & Cache:** DBs Gerenciados (ex. Atlas/Redis Cloud) ou Cloud SQL.

### Categoria 2: Gateway de Pagamento (Pix)

Temos duas opções principais para processamento de pagamentos Pix, ambas oferecendo taxas idênticas por transação, mas diferindo em acessibilidade.

**Opção A: Banco EFI**

- **Taxa Pix:** **1,19%** por transação recebida via API.
- **Status:** Totalmente acessível para empresas brasileiras.

**Opção B: Stripe**

- **Taxa Pix:** **1,19%** por transação recebida via API.
- **Status:** Atualmente disponível apenas para convidados (invite-only) no Brasil, exigindo aprovação.

### Categoria 3: Domínio e DNS

- **Custo:** Custo Anual Padrão (Registro e manutenção do domínio). O domínio do projeto será adquirido e gerenciado diretamente através da **Cloudflare** para resolução de DNS segura e roteamento de proxy.

### Conclusão Financeira

Ambas as arquiteturas garantem previsibilidade dos custos de infraestrutura. A opção Hetzner fornece um custo mensal fixo e extremamente baixo com performance consistente, enquanto a abordagem Cloud Run minimiza os custos durante períodos ociosos ao ligar e desligar a instância conforme necessário. Para pagamentos, tanto o Banco EFI quanto a Stripe oferecem taxas competitivas de 1,19% para Pix, embora o Banco EFI seja mais imediatamente acessível.

## 5. Foco Estratégico: Maestria em Instância Única

Nossa prioridade atual é a **entrega perfeita de uma instância comunitária independente (`@elo-instance/*`)**. O desenvolvimento do SaaS começará apenas após a aplicação principal estar totalmente operacional.

### A Visão SaaS de Longo Prazo

Embora o foco seja a instância, o objetivo de longo prazo é evoluir para uma Plataforma SaaS Multi-tenant:

- **Escopo do Portal (`@elo-portal/web` & `@elo-portal/api`)**: Um ponto de entrada global único para onboarding e descoberta.
- **Múltiplos Pares de Instância (`@elo-instance/api` & `@elo-instance/web`)**: Cada comunidade recebe seu próprio par de ambiente isolado.
- **Bibliotecas Core (`@elo-instance/core` & `@elo-portal/core`)**: Fonte única de verdade (SSOT) específica de domínio para cada escopo.

#### Principais Funcionalidades SaaS

- **Isolamento de Tenant:** Dados e configurações independentes para cada instância de comunidade.
- **Autogestão do Produtor:** Gestão direta de estoque e preços para os produtores.
- **Gestão de Assinaturas:** Níveis de serviço gerenciados via `@elo-portal/api`.

#### Evolução Arquitetural

- **Implantação Multi-Instância:** Criação automatizada de containers `@elo-instance/*` por tenant.
- **Roteamento Dinâmico:** Load balancer global roteando usuários para sua instância de comunidade específica.

## 6. Início Rápido

Certifique-se de ter o **Node.js 22+**, **PNPM 11+** e o **Docker** instalados.

1.  **Instalar Dependências:**

    ```bash
    pnpm install
    ```

2.  **Configurar Arquivos de Ambiente:**
    Cada contexto delimitado e aplicação possui seu próprio arquivo de ambiente. Copie cada `.env.*.example` para o seu respectivo arquivo sem a extensão `.example` e preencha com seus valores locais:

    ```bash
    # Orquestração da Instance (para o compose)
    cp instance/infrastructure/docker/.env.example instance/infrastructure/docker/.env

    # Runtime da API da Instance
    cp instance/services/api/.env.dev.example instance/services/api/.env.dev

    # Variáveis de build da Web da Instance
    cp instance/services/web/.env.dev.example instance/services/web/.env.dev

    # Configuração local dos bancos de dados
    cp instance/services/mongodb/.env.example instance/services/mongodb/.env.dev
    cp instance/services/redis/.env.example instance/services/redis/.env.dev
    ```

    Repita para `portal/` quando estiver trabalhando no contexto de Portal.

3.  **Inicie a Infraestrutura Local e Aplicações:**
    Nós fornecemos scripts de orquestração unificados via Turborepo para subir containers Docker e servidores de desenvolvimento local simultaneamente.

    Consulte a [Referência de Comandos](./commands-reference.mdx) para instruções sobre como iniciar a stack (`pnpm instance:dev`), executar validações ou realizar deploys para produção.
